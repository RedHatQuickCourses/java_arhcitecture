In object-oriented programming, an object is a fundamental building block. It acts as a self-contained entity that encapsulates both data and the behaviors that operate on that data. To understand how Java applications function, we must first look at the internal structure of an object, which is divided into two key components: Data, represented by instance fields or attributes, and Responsibilities, represented by methods and behaviors.

Let’s start by looking at data. Every object holds specific information that defines its current state. This data is stored in instance fields, which are variables declared within a class but outside of any specific method. Each individual object created from a class carries its own copy of these fields, allowing different objects of the same class to maintain unique states. For example, consider a Car object. Its instance fields might include color, make, model, and current speed. While every car has these attributes, one specific car object might be a red Toyota Camry traveling at sixty miles per hour, while another is a blue Honda Civic traveling at forty-five. Those distinct values define the state of each unique object.

Next, we have responsibilities, which are the actions an object can perform or the services it can provide. These are implemented through methods. You can think of methods as the "task performers" within an object. They contain the logic that defines how an object behaves, how it interacts with its own data, and how it communicates with other parts of your application. Using our Car example, the responsibilities—or methods—might include starting the engine, accelerating, braking, or changing gears. When you call the accelerate method, it directly modifies the internal current speed data of that specific car object.

The true power of this structure lies in the symbiotic relationship between data and methods. Methods are designed to operate on the object’s own internal fields, ensuring that the object's state is managed consistently. This bundling of data and logic is the core principle of encapsulation. By keeping the internal workings hidden and interacting only through well-defined methods, you create software that is more modular, maintainable, and reusable.

Now, let's put this into practice with a hands-on activity. We will define a simple Java class to see how these concepts work in code.

For our first step, create a Java file named Book.java. This class represents a book object. Inside, you will declare instance fields for the title, author, page count, and availability status. We will also include a constructor to initialize these fields when a new book is created, along with methods to display book information, borrow a book, and return a book. These methods will directly modify the book's availability status.

For our second step, create a file named LibraryApp.java. This is our main class where we will create actual Book objects. In this file, we will instantiate two different books, display their initial states, and then invoke our borrow and return methods to demonstrate how each object maintains its own independent state.

For our final step, it is time to compile and run the code. Open your terminal or command prompt, navigate to the folder where you saved your files, and run the Java compiler by typing "javac Book.java LibraryApp.java". Once compiled, run the application by typing "java LibraryApp".

When you run this program, you will see the console output confirm that each book object maintains its own unique data. You will see the state of each book change as the methods interact with the internal fields—for instance, marking a book as unavailable when borrowed and available again when returned.

This activity perfectly illustrates that instance fields constitute the object's data, while methods define its responsibilities. By seeing how book one and book two operate independently, you can clearly see how objects function as self-contained, encapsulated entities in your Java applications.
